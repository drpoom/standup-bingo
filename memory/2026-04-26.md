## v5.2.1 Critical Bug — Spawning Broken (11:36 AM)

### Reported
User (John) reports that obstacles, coins, and items are NOT spawning after game start in v5.2.1.

### Investigation Status
- `resetStage()` sets `lastSpawnTime = -2` and `gameDuration = 0`
- `animate()` has a grace period: `const spawnGraceActive = gameDuration < 1.5`
- During countdown, `countdownLocked = true` → animate returns early, `clock.getDelta()` is consumed but `clock.getElapsedTime()` doesn't advance
- After countdown ends: `countdownLocked = false`, game loop runs
- `gameDuration` is never incremented! It stays at 0 forever → `spawnGraceActive` is always `true` → spawn never fires
- **Root cause:** The `gameDuration += delta;` line was likely removed or broken during the v5.2.1 changes

### Next Steps
1. Find and fix `gameDuration` increment in the animate loop
2. Add automated Playwright test to verify spawning works for all stages
3. Deploy fix + test

# 2026-04-26 — Elango Surfers Playwright Test Debugging

## Bug: Obstacles/Coins Never Spawn After Debug Stage Jumps

### Root Cause
Debug keys `1/2/3` call `resetStage()` without triggering `startStageCountdown()`. This caused:

1. **Spawn Grace Period Block**: `gameDuration = 0` → `spawnGraceActive = true` for ~1.5s, blocking all obstacle/coin spawns
2. **No Visual Feedback**: No countdown or invincibility shield to indicate game resuming
3. **countdownLocked State**: If stage jump occurred during initial countdown, `countdownLocked` remained `true`, blocking `animate()` game loop

### Files Involved
- `src/App.vue`: `resetStage()` at line ~4964, `animate()` at line ~3278, `handleKeyDown()` debug sequence at line ~4884
- `tests/spawn-after-restart.spec.ts`: Test verifying spawns after stage jumps
- `tests/cobblestone-regression.spec.ts`: Keyboard timeout tests

### Fix Applied
Updated `resetStage()` to skip grace period for stage jumps:

```js
// Before (broken):
gameDuration = 0;
lastSpawnTime = -2;

// After (fixed):
gameDuration = 1.5; // skips spawn grace period
lastSpawnTime = clock.getElapsedTime() - spawnInterval; // spawns next frame
countdownLocked = false; // ensures game loop runs
```

### Lessons Learned
- **Grace period logic**: `spawnGraceActive = gameDuration < 1.5` is meant for initial game start only, not stage transitions
- **Debug paths need same treatment as production paths**: Stage jumps via debug keys should mirror boss-defeat stage transitions
- **countdownLocked must be cleared**: Any manual reset must explicitly clear `countdownLocked` to prevent game loop deadlock
- **Archie (glm-5.1:cloud) for deep analysis**: 2m14s runtime, thorough code path tracing

### Reported
User reports Stage 2 textures (cobblestone road, Fachwerkhaus buildings) not applied properly. Test didn't catch it.

### Root Cause
`applyStageVisuals()` is called synchronously in `resetStage()`, but texture loading is async:
- `loadTexture()` returns immediately, loads in callback
- Game renders before textures load → wrong visuals
- No visual verification in tests (only checked stage indicator text)

### Fix Applied
1. **Game**: Added retry mechanism in `applyStageVisuals()` - if texture not loaded, wait and apply in callback with `needsUpdate = true`
2. **Test**: Added visual verification - check for cobblestone texture on road mesh via `window.__getRoadTexture()` helper
